Graphes dynamiques ?

Quel type de dynamique ?

  • des valeurs associées aux sommets et arêtes ?
  • apparition et disparition de sommets ou arêtes ?

Le problème avec les algorithmes :

  • Dès qu'un résultat est calculé, il est invalidé par la dynamique.
  • Peut-on stopper la dynamique et recalculer ?
  • Tout dépend du modèle de dynamique...

Images de graphes

Chaque évolution du graphe est considérée comme un graphe à part entière.

Pratique, les algorithmes classiques de la théorie des graphes sont utilisables

Problème il faut recalculer les solutions pour chaque changement.

Méthodes agrégatives

L'évolution complète du graphes est connue par avance, le graphe dynamique est agrégé en un graphe statique.

Cela nous permet aussi d'utiliser les algorithmes classiques de la théorie des graphes.

Mais il faut connaître l'intégralité du graphe par avance.

Ré-optimisation

On construit et maintient des structures dans le graphe dynamique (e.g. un arbre couvrant)

La mise-à-jour de la structure existante après modifications est souvent moins coûteuse que de tout recalculer de zéro.

Une bibliothèque Java avec une API publique pratique

  Graph g = new SingleGraph("MyGraph");
  g.read("some-file.tlp");
  g.getDegree();
  g.display();

Basée sur un modèle événementiel : les sommets et les arêtes sont ajoutés, retirés, modifiés.

Interaction avec d'autres outils :

  • Offline : plusieurs formats d'import / export de fichier
  • Online : à travers une API de connexion réseau

Le modèle événementiel

La dynamique du graphe est exprimée par une flux d'événements

Ces derniers peuvent être :

  • Addition ou retrait de sommets.
  • Addition ou retrait d'arêtes.
  • Addition, mise-à-jour ou retrait d'attributs associés aux sommets, arêtes ou au graphe.
  • Étape temporelle.

Le flux d'événements modifie l'état d'un graphe.

Modèle événementiel

Sources

Produisent un flux d'événements.

Sinks

Reçoivent un flux d'événements et le traitent..

Pipes

Sont à la fois une source et un sink. Un graphe est un pipe.

Sink

Pipelining

Les sources envoient leurs événements vers des sinks.

  • Design pattern Observer
  • Publish / Subscribe
  • Listeners Java Swing

Les sources, pipes et sinks peuvent être connectés en un pipeline d'éléments.

Exemple d'un pipeline très courant :

Pipelining

Le flux d'événements peut transiter entre les sources et les sinks :

  • à travers le réseau,
  • entre processus,
  • entre threads.

Par exemple, la visualisation s'éxécute dans un thread distinct, mais peut aussi s'exécuter sur une autre machine, alors que la simulation et le graphe principal du modèle fonctionnent sur une autre.

Pipelining

Recevoir des événements d'un autre processus/thread/machine :

Graph g = new SingleGraph("RWP");
ThreadProxyPipe pipe = getPipeFromElsewhere(); //fake function
pipe.addSink(g);
g.display(false);

while (true) {
    pipe.pump();
    Thread.sleep(1);
}

Graphes

Plusieurs classes permettent de modéliser différentes structures de graphe.

  • "single" graph (1-graph), "multigraph" (p-graphes, ayant potentiellement plusieurs arêtes entre deux nœuds).
  • Graphes dirigés, non dirigés...

Représentation d'un graphe à un temps donné, mais cette représentation est évolutive.

Attributs de données

  • Une quantité arbitraire d'attributs peut être associée à n'importe quel élément du graphe ou au graphe lui-même.
  • Un attribut est composé d'un identifiant et d'une valeur qui peut-être n'importe quel objet Java.
g.addAttribute("My attribute", aValue);
Node n = g.getNode("A");
n.setAttribute("xy", 23.4, 55.0);
Edge e = g.getEdge("AB");
e.removeAttribute("selected");
double w = e.getNumber("weight");

      .networksimplex.DynamicOneToAllShortestPath;

Interactions Online : NetStream

  • Exporte un flux d'événements à d'autres applications / machines / langages.
  • Bidirectionnel, à partir et vers GS.
  • Protocole réseau binaire.
  • Implementation TCP socket (et WebSocket).
  • Mutli-langages (Java, C++, Python, JS, ...)
import org.graphstream.stream.netstream.NetStreamReceiver;
//...
NetStreamReceiver net = new NetStreamReceiver(2001);
ThreadProxyPipe pipe = net.getDefaultStream();
pipe.addSink(graph);
while (true) {
	pipe.pump();
	Thread.sleep(100);
}

Extension NetLogo

  • Les agents NetLogo (tortues, liens et observateur) envoient des événements graphe à une application externe.
  • Cette dernière maintient un graphe dynamique et peut appliquer les algorithmes et outils de GraphStream sur celui-ci.
  • Les résultats sont retournés à NetLogo.
  • Les agents peuvent les recevoir et les utiliser pour prendre leurs décisions.
